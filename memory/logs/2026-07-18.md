# 记忆日志 — 2026-07-18

## 会话主题

搭建个人 Agent 工具箱 `agent-toolkit/`，核心原则：**极简、模块化、低耦合**。

## 当前在做什么事

正在实现一条从「用户原话」到「可验收结果」的完整编程代理管线：

```text
router → intake → refine → prompt_optimize → dispatch → execute → prune-check → accept
```

目前已完成前 5 个节点 + prune-check + **acceptance.py** + Headroom 集成，七阶段管线全部跑通。

## 关键决策与偏好

### 架构原则

- **一个功能一个文件**，不要为了低耦合强行拆分。
- **数据、逻辑、配置分离**。
- 工作流七阶段固定，但最外层加了 `router` 分流。

### 模型选择偏好

- **速度优先**，成本敏感。
- intake 不用 LLM；refine / prompt_optimize / dispatch / execute / router 默认用 `deepseek-v4-flash`。
- **不用 `deepseek-v4-pro`**，宁愿用 5.6 系列（虽然慢）。
- packyapi 的 `gpt-5.6-luna/terra/sol` 便宜但延迟高，只留给 quality gate / acceptance 场景。
- 本地小模型优先用于代码生成、敏感数据、高频小任务。

### 执行节点偏好

- 弱模型执行任务，但强模型写任务时必须给出 **输入、输出、验收标准**。
- 任务跑完后模型自己自检一遍，通过了才算完工。
- 自检有问题要返工，最多 2 次；还有问题直接抛异常，请强模型诊断。
- 每个任务上下文要隔离，写坏了可以重放该节点。
- 支持 `--apply` 实际写文件，写之前自动备份 `<file>.execute.bak`。

### 省钱叠加策略

1. `router` 分流：非编程任务不进完整管线。
2. `dispatch` 路由：简单单文件任务不调强模型规划。
3. `Headroom` 代理：所有 LLM 调用自动压缩上下文。

## 待讨论点

### 全量可回溯记忆

用户提出想保留**全量可回溯记忆**。当前先用按日期命名的 Markdown 日志记录关键决策和偏好，但是否要：

- 每次对话都存入结构化数据库（SQLite）？
- 记忆分 `current`（当前有效）和 `archive`（历史参考）两层？
- 旧记忆如何衰减权重，避免干扰当前决策？
- 是否需要向量检索，支持按语义召回历史讨论？

**结论**：先按日期日志记录，全量可回溯记忆的实现方式待后续专门讨论。

### 记忆/偏好如何注入工作流

用户的记忆和偏好（如速度优先、不用 v4-pro、返工 2 次、极简原则等）需要在某个管线环节作为背景注入，否则每次都要重新声明。待讨论：

- 注入位置：
  - `intake` 之后？作为上下文跟随整个管线。
  - `refine` 阶段？把偏好转成约束条件。
  - `prompt_optimize` 阶段？直接写入编程代理 prompt。
  - `dispatch` 阶段？影响模型选择和任务拆分策略。
- 注入方式：
  - 静态读取 `memory/logs/` 最新几条？
  - 动态检索与当前任务相关的记忆？
  - 只注入 `current` 记忆，还是也注入衰减后的 `archive`？
- 优先级：哪些偏好必须遵守，哪些只是参考？

**结论**：记忆注入的位置、方式和权重规则待后续专门讨论。
